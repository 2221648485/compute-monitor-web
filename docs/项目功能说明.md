# ComputeMonitor 项目功能说明

## 项目定位

ComputeMonitor 是面向 Kubernetes GPU 集群的算力监控与资源管理平台。后端基于 Go/Gin，统一纳管多集群配置，周期同步 Kubernetes 资源，接入 Prometheus 指标，并提供节点、Pod、Deployment、GPU、告警、审计和迁移任务相关 API。

平台不是单纯的接口展示工具。它的核心价值是把“集群接入、资源同步、算力观测、工作负载查看、告警审计、迁移任务”组织成运维闭环。

## 后端能力

- 认证与用户：JWT access token、refresh token、Redis session、管理员角色校验。
- 集群管理：创建、查询、更新、删除、连接测试 Kubernetes 集群配置。
- K8s 同步：同步 Namespace、Node、Pod、Deployment、Service 到 MySQL 缓存。
- 节点与工作负载：查询节点、节点详情、节点 Pod、Pod、日志、Deployment、Service。
- 指标监控：通过 Prometheus 查询 CPU、内存、磁盘、网络和 GPU 时间序列。
- GPU 资源：展示 GPU 列表、节点 GPU、GPU 汇总和使用率信息。
- 多维资源感知：提供算力、网络、存储、能源和跨集群 overview。
- 迁移管理：迁移计划、可行性评估、目标优化、调度和任务生命周期。
- 告警审计：告警规则、告警事件、审计日志。
- 兼容接口：`/api/v2` 面向旧调度器或旧前端，普通控制台不以它作为主操作面。

## 前端信息架构

新前端采用 Vue3 + Vite + TypeScript，以真实接口调用为基础，不使用 mock 数据。

- 总览：展示集群数、CPU/GPU 总量、当前集群负载和跨集群资源概览。
- 集群：展示已纳管集群，提供连接测试和手动同步入口。
- 节点/GPU：合并节点和 GPU 资源，避免把相似监控对象拆成重复页面。
- 工作负载：聚合 Pod 与 Deployment，用于快速判断运行状态。
- 迁移：轻量展示迁移计划和任务，不提前承诺复杂决策工作台。
- 告警审计：合并告警事件、告警规则和审计日志。
- 用户/RBAC：接入当前后端真实用户管理能力，支持用户列表、新增、编辑、启停、重置密码；完整 RBAC 需要后端继续实现角色表、权限表、角色权限绑定和权限码鉴权。

## API 对接约定

后台主接口前缀为 `/api/admin`。登录与刷新 token 是公开接口，其余后台接口需要请求头：

```text
Authorization: Bearer <access_token>
```

后端统一响应格式：

```json
{ "code": 0, "message": "success", "data": {} }
```

分页参数为 `page`、`size`，分页响应为：

```json
{ "items": [], "total": 0, "page": 1, "size": 20 }
```

开发期后端未配置 CORS 时，前端通过 Vite proxy 访问 `/api`。

## 设计取向

审美参考 `awesome-design-md` 中 NVIDIA 风格：黑白灰结构、绿色强调、2px 小圆角、无渐变、无装饰阴影、密集工程数据布局。该方向与 GPU/算力监控平台的语义最匹配。

## 当前边界

- 不做接口数量统计页。
- 不把 `/api/v2` 兼容接口作为普通用户主页面。
- 不使用前端硬编码模拟数据。
- 不伪造完整 RBAC 配置 UI；当前后端主要是 admin-only，`operator/viewer` 只是用户模型中的角色字段。
- Network、Storage、Energy 等资源感知数据来源需在后续联调中验证真实性。
